# 软件开发生存期
## 软件开发模型
**瀑布模型**：go 书城项目属于典型的 “功能型 Web 应用”，核心需求边界清晰 —— 即围绕 “书籍展示、检索、购买、用户管理、订单履约” 等标准化电商模块展开，且需求在招标阶段已通过招标文件明确、，不存在大量模糊或易变更的需求点。  
符合瀑布模型的优势： “需求先行、阶段固化”，需在项目初期明确全部需求并拆解为可执行的阶段目标。对于 go 书城这类需求稳定的项目，可直接按 “需求分析→设计→开发→测试→部署” 的线性流程推进，避免因需求频繁变动导致的流程反复，减少开发资源浪费
## 软件项目组织
**组织目标**开发一个功能完善的在线书城系统，支持图书浏览、购买、订单管理、用户管理等核心功能，为用户提供便捷的购书体验，为管理员提供高效的后台管理功能。

### 需求调研与分析阶段
**目标**：明确用户需求与业务边界，为后续的软件开发提供准确且全面的依据。

#### 具体过程
1. **准备工作**
    - **组建团队**：成立由产品经理、市场调研人员、部分开发和测试人员组成的需求调研小组。产品经理主导整个需求调研与分析过程，市场调研人员负责收集市场数据，开发和测试人员从技术实现和测试角度提供早期意见。
    - **确定范围**：初步确定调研的业务领域和目标用户群体，例如针对电商类软件，可能包括B端商家用户和C端消费者用户。
2. **收集信息**
    - **市场调研**：通过查阅行业报告、分析竞争对手产品、研究市场趋势等方式，了解行业现状和市场需求。例如，分析其他主流电商平台的功能特点、用户评价、市场占有率等，找出差异化竞争点。
    - **业务方沟通**：与公司内部相关业务部门（如销售、运营等）沟通，了解业务流程和业务需求。
3. **需求整理与分析**
    - **需求分类**：将收集到的需求按照功能模块（如商品管理、订单管理、用户管理等）、业务流程（如购物流程、支付流程等）或用户类型进行分类整理。
    - **需求筛选与优先级确定**：对需求进行评估，考虑需求的必要性、可行性、商业价值等因素，筛选出合理的需求，并确定优先级。例如，对于电商软件，用户注册登录和商品展示功能属于基本需求，优先级较高；而一些个性化的社交分享功能，优先级可以相对低一些。
    - **需求细化**：对每个需求进行详细描述，包括需求背景、功能描述、业务规则、输入输出要求等。比如，对于商品搜索功能，详细说明搜索的关键词匹配规则、搜索结果的排序方式等。
4. **文档撰写**
    - **编写需求规格说明书（SRS）**：以正式的文档形式记录软件的功能需求、非功能需求（如性能、安全性、可维护性等）、接口需求等内容，为后续的开发、测试等工作提供准确的依据。
    - **生成产品需求文档（PRD）**：侧重于从产品角度对功能进行详细描述，包括产品的功能结构、页面交互设计、业务流程说明等，帮助团队成员更好地理解产品的功能和用户体验要求。
5. **需求评审**
    - **组织评审会议**：召集开发团队、测试团队、业务部门、客户等相关人员，对需求文档进行评审。
    - **评审内容**：检查需求的完整性、准确性、一致性、可行性等，提出问题和修改建议。例如，开发人员可能会从技术实现难度角度提出某些需求无法实现或需要调整，测试人员则会关注需求是否可测试。
    - **需求确认**：根据评审意见对需求文档进行修改完善，直到各方达成共识，最终确定需求基线。

### 概要设计阶段
**目标**：确定系统架构与技术方案，为详细设计和编码开发提供整体框架和指导。

#### 具体过程
1. **需求理解与分析**
    - **需求研读**：开发团队成员深入研读需求规格说明书（SRS）和产品需求文档（PRD），全面理解软件的功能需求、性能要求、业务规则等。
    - **需求澄清**：对于不明确或有疑问的需求，与产品经理、需求调研小组进行沟通，确保对需求的准确理解。
2. **架构设计**
    - **确定架构风格**：根据需求特点和业务场景，选择合适的软件架构风格，如分层架构（表现层、业务逻辑层、数据访问层等）、微服务架构、单体架构等。
    - **设计系统拓扑结构**：规划系统中各个组件、模块之间的关系和交互方式，绘制系统架构图。
    - **考虑非功能需求**：在架构设计中充分考虑性能、安全性、可扩展性、可维护性等非功能需求。
    - **评估技术组件**：根据架构设计和需求，对实现软件所需的技术组件（如编程语言、数据库、框架、中间件等）进行评估和选择。
    - **技术可行性分析**：对选定的技术进行可行性分析，包括技术的成熟度、团队成员的技术掌握程度、技术的学习成本等。确保选择的技术能够在项目周期内被有效应用和实施。
    - **确定技术方案**：综合考虑各种因素，确定最终的技术选型方案，并编写技术选型报告，详细说明选择的技术组件及其原因、技术架构的实现思路等。
3. **文档编写**
    - **撰写架构设计文档（AD）**：记录系统的架构设计，包括架构风格、拓扑结构、组件划分、接口定义、数据流程等内容，为详细设计和编码开发提供具体的指导。
    - **完善技术选型报告**：将技术选型过程中的评估结果、分析结论等内容整理到技术选型报告中，作为项目技术决策的重要依据。
4. **设计评审**
    - **组织评审会议**：邀请公司内部的技术专家、架构师、开发团队成员、测试团队成员等对架构设计文档和技术选型报告进行评审。
    - **评审内容**：检查架构设计的合理性、技术选型的可行性、是否满足需求等。例如，技术专家可能会从技术发展趋势角度提出架构设计的改进建议，测试人员会关注架构设计对测试工作的影响。
    - **设计确认**：根据评审意见对架构设计和技术选型进行调整和优化，直至满足项目要求，最终确定概要设计基线。
